# Epic EP-001: Core Authentication Flows

Status: draft

## Goal

Users can establish individual accounts, log in with personal credentials, maintain a persistent session, and reset their password through admin-mediated one-time-link flows — with no shared credentials and no system email infrastructure.

## Context

OHM's current model gives all musicians a single shared login that exposes personal data, creating a security risk and a GDPR liability. EP-001 replaces it: admins invite musicians via one-time invite links (generated and displayed in the admin UI, sent manually), musicians activate their accounts through a form, and server-side sessions gate access across the application. Password recovery follows the same admin-mediated pattern. A CLI bootstrap command enables the first admin account to be created before any invite flow is possible.

## Personas

- **Admin**: An association member with management privileges. Works via a desktop browser. Generates invite and reset links, copies them from the admin UI, and sends them manually (e.g. from the association's email address). Non-technical: does not interact with the server directly.
- **Musician**: An association member. Receives invite or reset links from the admin via email. Activates or recovers their account on a single-page form. Not necessarily technical.
- **System Bootstrapper**: A technical operator performing first deployment. Runs a CLI command to create the initial admin account before any musician accounts exist. Has SSH access to the server.

### Completion Criteria

An admin can generate invite links and password-reset links for musician accounts, see them displayed in the admin UI for copying, and regenerate them when expired. A musician can activate a pending account and reset a password via a single-page form using the provided link. Login with valid active-account credentials creates a persistent session; logout destroys it. Unauthenticated requests to protected routes redirect to the login page. A system bootstrapper can create the first admin account via CLI without going through the invite flow.

### Non-Functional Requirements

- NFR-000-1: Session cookies must be HttpOnly, Secure, and SameSite=Strict.
- NFR-000-2: Passwords must be hashed with Argon2id; never stored in plain text and never retrievable.
- NFR-000-3: Invite and password-reset tokens must be generated from a CSPRNG: 32 random bytes, base64url-encoded.
- NFR-000-4: Session lifetime is 7 days rolling.
- NFR-000-5: All state-changing HTTP requests (POST, PUT, PATCH, DELETE) must be CSRF-protected.
- NFR-000-6: All UI text, error messages, and labels must be in French.

### Related External Components

None. Authentication is entirely internal; no external identity provider or email service is used.

### Out of Scope

- **Musician record fields** (name, instrument, birth date, phone, address) — ep-2 scope; account creation UI is initiated there.
- **Phone/address consent implications** (what consent enables for admin editing, admin consent withdrawal on behalf of musician) — ep-4 scope.
- **Account anonymization flow** — ep-4 scope. Session invalidation triggered by anonymization is owned there.
- **Admin role management and last-admin protection** — EP-002 (this phase).
- **Sheet music access display** — EP-002 (this phase).
- **Self-service account creation** — no self-registration; admin-mediated invite flow only.
- **System email infrastructure** — no emails sent by the system; links are sent manually by admins.
- **Public homepage** — ep-4 scope.
- **GDPR processing restriction flag** — ep-4 scope.

### Assumptions

- **ASM-000-1**: Login uses email as the identifier. *Why*: Email is the unique non-anonymized identifier on the Account entity per the information model. — ⚠️ Confidence: HIGH
- **ASM-000-2**: The privacy notice acknowledgement checkbox on the invite form is required to submit; the phone/address consent checkbox is optional. *Why*: Privacy notice acknowledgement is a legal compliance requirement; the spec lists both fields without mandating either, but consent cannot be revoked once given on this form, while the privacy notice protects the association legally. — ⚠️ Confidence: HIGH
- **ASM-000-3**: The bootstrap CLI creates an account in `active` state with the admin role already granted, bypassing the invite flow. *Why*: The configuration spec states this explicitly: "Creates an account in active state and grants it the admin role." — ⚠️ Confidence: HIGH

## Capability CAP-001 - Account Invitation and Activation

Admin can invite a new musician to the platform via a one-time link; the musician follows it to complete account setup and gain access.

### References

- Functional spec: `specs/functional-specs/01-account-lifecycle.md` — Invite Flow section
- Functional spec: `specs/functional-specs/00-information-model.md` — InviteToken entity
- Technical spec: `specs/technical-specs/01-auth-and-security.md` — Token Generation, Public Routes
- Technical spec: `specs/technical-specs/04-routing.md` — `/invitation/{token}` routes

### Open Questions

- **OQ-001-1**: The auth-security spec (`specs/technical-specs/01-auth-and-security.md`, Public Routes table) lists the invite route as `/invite/:token`, while the routing spec (`specs/technical-specs/04-routing.md`) lists it as `/invitation/{token}`. These are inconsistent. — *Impact if unresolved*: Story Writers will encounter two different paths and cannot determine the canonical route without consulting additional material. The routing spec is the authoritative source for path definitions and should be treated as correct; the auth-security spec's Public Routes table contains a stale path and should be updated in a separate task.

### Description

When an admin creates a musician account (account creation UI is ep-2 scope), the system generates an InviteToken and displays the invite URL in the admin UI so the admin can copy and send it manually. The musician follows the link and is presented with a form containing a password field, a privacy notice acknowledgement checkbox, and a phone/address consent checkbox. Submitting the form atomically transitions the account from `pending` to `active`, sets the password and consent flag, marks the token used, and logs the musician in. Expired or already-used invite links display an informative message; the admin can generate a replacement from the account detail view at any time.

None. This is the first capability in the authentication flow; account creation (which triggers token generation) is an ep-2 concern but produces the token that this capability acts on.

### Out of Scope

## Capability CAP-002 - Login, Logout, and Session Management

Musicians and admins can log in with their email and password, maintain a persistent session across requests, and log out, with unauthorized access to protected routes blocked.

### References

- Technical spec: `specs/technical-specs/01-auth-and-security.md` — Session-Based Authentication, Password Hashing, Public Routes, Role Checks
- Technical spec: `specs/technical-specs/02-configuration.md` — Bootstrap: First Admin Account, Emergency Admin Recovery
- Technical spec: `specs/technical-specs/04-routing.md` — Public routes (`/connexion`, `/deconnexion`), Authenticated route group, middleware stack
- Technical ADR: `specs/technical-adrs/001-authentication.md` — Session-side sessions rationale
- Functional spec: `specs/functional-specs/01-account-lifecycle.md` — Account States (only `active` accounts may log in)

### Description

An `active` account can authenticate by submitting email and password on the login form. On success, a server-side session is created and a signed cookie is set; the session is valid for 7 days rolling. Protected routes verify a valid active-account session on every request; unauthenticated requests redirect to the login page. Logout destroys the session server-side and redirects to the login page. A CLI command (`mise run seed-admin`) lets a technical operator create the first admin account when none exists yet, bypassing the invite flow. A second CLI command (`mise run recover-admin`) provides emergency password recovery when the last admin cannot log in.

| TBD | Login is rejected for pending and anonymized accounts with an informative message | P1 | Depends on login story |
| TBD | Unauthenticated access to a protected route redirects to the login page | P1 | Depends on session creation story |
| TBD | Musician logs out and the session is destroyed | P1 | Depends on login story |
| TBD | System bootstrapper creates the first admin account via CLI | P1 | No dependency on web flow stories; requires an empty admin table |

### Depends on

None. The bootstrap CLI creates an `active` admin account directly without going through the invite flow; all login and session stories can be built and tested independently. CAP-001 (invite flow) depends on this capability — not the reverse.

### Out of Scope

- Session invalidation on account anonymization — ep-4 scope (triggered by the anonymization action).
- Admin role verification on admin-only routes — EP-002 CAP-001 (this phase).
- Password reset — CAP-003 (this epic).

### Assumptions

- **ASM-002-1**: The login page is accessible without authentication (no redirect loop). *Why*: The technical spec's public routes table lists `/connexion` explicitly. — ⚠️ Confidence: HIGH
- **ASM-002-2**: The bootstrap CLI refuses to run if an active admin account already exists, to prevent duplication. *Why*: The configuration spec states this explicitly. — ⚠️ Confidence: HIGH
- **ASM-002-3**: The emergency recovery CLI accepts an email and new password, verifies the account is `active`, and updates the password hash without checking the admin role. *Why*: The configuration spec states this explicitly: "Does not check for admin role — this is intentional." — ⚠️ Confidence: HIGH

## Capability CAP-003 - Password Reset

Admin can generate a one-time password-reset link for an active account; the musician follows it to set a new password.

### References

- Functional spec: `specs/functional-specs/01-account-lifecycle.md` — Password Reset section
- Functional spec: `specs/functional-specs/00-information-model.md` — PasswordResetToken entity
- Technical spec: `specs/technical-specs/01-auth-and-security.md` — Token Generation, Password Hashing, Public Routes
- Technical spec: `specs/technical-specs/04-routing.md` — `/reinitialiser-mot-de-passe/{token}` routes

### Description

An admin navigates to a musician's account detail view and generates a PasswordResetToken. The reset URL is displayed in the admin UI for copying and manual sending. The token expires after 7 days. The musician follows the reset link and submits a new password on a form; the account's password hash is updated atomically and the token is marked used. Expired or already-used reset links display an informative message; the admin can generate a replacement at any time. Password reset is only available for `active` accounts; the operation is blocked for `pending` and `anonymized` accounts.

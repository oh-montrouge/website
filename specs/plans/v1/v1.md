# V1 Roadmap

Sequenced delivery plan for `specs/goals/vision.md` ┬¦ V1.

---

## Phase 0 ŌĆö Walking Skeleton

**Goal:** full vertical slice working locally. One DB table rendered in a browser. Proves
the stack (Go, Buffalo, Pop, PostgreSQL, Docker, Mise) hangs together before any feature work.

- Buffalo project initialised (`app.go`, `go.mod`, directory layout)
- Docker Compose: `app` + `postgres` containers, source directory mounted for live reload
- `mise.toml` with `dev` and `migrate` tasks
- `.env.development.example` (placeholders for `DATABASE_URL`, `SESSION_SECRET`, `GO_ENV`)
- First migration: `instruments` table (seeded, no FK dependencies)
- Instrument seed migration (OHM Agenda controlled list + "Chef d'orchestre")
- One public route `/instruments` ŌåÆ handler ŌåÆ Plush template listing instrument names
- `buffalo dev` starts, page loads data from DB

---

## Phase 1 ŌĆö CI

**Goal:** automated quality gate on every push; test infra established for all subsequent work.

- Lefthook (`lefthook.yml`): `gofmt`, `go vet`, `golangci-lint --fast` on commit
- `testcontainers-go` harness: `TestMain` spins up a PostgreSQL container, applies
  migrations, runs all tests in that binary, tears down
- One integration test for the Phase 0 handler (verifies instruments are returned from DB)
- GitHub Actions: `lint` job + `test` job running in parallel; both required to pass
- Coverage report attached to the test job (no hard gate in V1, informational)

---

## Phase 2 ŌĆö Production Infra

**Goal:** application reachable at the OVH domain over HTTPS, daily backups running.

- Multi-stage `Dockerfile` (build stage: `golang:alpine`; runtime stage: `alpine`, ~25ŌĆō30 MB image)
- `docker-compose.yml` extended with `caddy` service; `Caddyfile` with automatic HTTPS
- `.env.production.example`
- `mise run deploy` task: SSH to VPS ŌåÆ `docker compose pull` ŌåÆ `up -d` ŌåÆ `buffalo pop migrate`
- `DEPLOY_HOST` read from operator shell profile (not committed)
- `scripts/backup.sh`: `pg_dump` ŌåÆ gzip ŌåÆ upload to OVH Object Storage (30-day retention)
- VPS cron setup instructions for the backup script
- Production smoke test checklist (HTTPS green, login page loads, backup job runs once manually)

---

## Phase 3 ŌĆö Features

Each section is an independently deployable unit. Implement in order ŌĆö each section
depends on the ones above it.

---

### 3.1 Auth Foundation

Everything behind a login requires this.

- `http_sessions` migration (custom schema with `account_id` column + index)
- `roles` + `account_roles` migrations; `admin` role seeded
- `accounts` migration (full schema per `technical-specs/00-data-model.md`)
- Login form + POST handler (Argon2id verify, session create with `account_id` write)
- Logout handler (session destroy, redirect to `/connexion`)
- `RequireActiveAccount` middleware
- `RequireAdmin` middleware (queries `account_roles` on every request)
- `seed-admin` grift task (`mise run seed-admin`)
- `recover-admin` grift task (`mise run recover-admin`)

---

### 3.2 Public Pages + Navigation Shell

Static pages and the navigation structure that wraps all subsequent features.

- Homepage template (placeholder copy; real copy provided by association before launch)
- Privacy notice template (content from `specs/goals/gdpr.md` ┬¦5)
- Layout template with auth-aware navigation:
  - Unauthenticated: login link
  - Authenticated: events, profile, sheet music (if configured), logout
- Footer with persistent privacy notice link (all pages)
- `SHEET_MUSIC_URL` env var wired to show/hide "Partitions" menu item

---

### 3.3 Account Lifecycle

Invite and password reset flows. Depends on 3.1.

- `invite_tokens` + `password_reset_tokens` migrations
- Invite URL display in admin UI (after account creation ŌĆö admin copies and sends manually)
- Invite form: password field, privacy acknowledgement checkbox, phone/address consent checkbox
- Invite flow completion (atomic transaction per `technical-specs/05-implementation-notes.md`):
  activate account, set password hash, set consent flag, clear phone/address if no consent,
  mark token used, seed RSVP records for future events, create session, redirect to `/evenements`
- Token regeneration (admin generates new invite link; invalidates existing token)
- Password reset: token generation + display, reset form, apply new password, mark token used
- Expired/used token handling (generic informative message, no account state change)

---

### 3.4 Musician Management

Full admin control over musician accounts, plus musician's own profile view. Depends on 3.3.

- Musician list (`/admin/musiciens`)
- Create musician account (with under-15 rule, invite token generated on save)
- Edit musician account (name, email, instrument, birth date, parental consent URI; phone/address gated on consent flag)
- Admin role grant / revoke (with last-admin protection check)
- Delete pending account (with last-admin protection check)
- Anonymization (atomic transaction: clear personal fields, remove roles, delete RSVPs,
  invalidate tokens, revoke session; last-admin protection check before transaction)
- Consent withdrawal: clear phone, address, and consent flag together
- Toggle processing-restricted flag (display only in V1)
- Musician's own profile view (`/profil`): name, instrument, email, birth date if set,
  phone/address if consented, static consent-withdrawal notice
- Retention review list (`/admin/retention`): accounts whose last-payment season ended > 5 years ago
- `seed-dev` grift task (implement once schema is complete enough to generate realistic data)

---

### 3.5 Season Management

Depends on 3.1.

- `seasons` migration
- Season list (`/admin/saisons`) with inline creation form
- Create season (label, start date, end date; not automatically designated current)
- Designate current season (atomic swap transaction; exactly-one invariant maintained)

---

### 3.6 Fee Payments

Depends on 3.4 + 3.5.

- `fee_payments` migration
- Record fee payment on musician detail page (duplicate guard: reject if (account, season) pair exists)
- Edit fee payment (`/admin/cotisations/{id}/modifier`)
- Delete fee payment
- First inscription date: derived on demand (`MIN(payment_date)`), shown in admin musician detail view

---

### 3.7 Events + RSVP

Depends on 3.4 (accounts must exist for RSVP seeding).

- `events`, `rsvps`, `event_fields`, `event_field_choices`, `rsvp_field_responses` migrations
- Event list for authenticated users (`/evenements`): past 30 days + all upcoming, own RSVP state shown
- Event detail page: full RSVP list; for concerts, instrument per `yes` RSVP; for `other`, field responses per `yes` RSVP
- RSVP update (`/evenements/{id}/rsvp`): yes/no/maybe; instrument selection for concerts (main instrument pre-selected); response clearing on transition away from yes
- Admin event management list (`/admin/evenements`)
- Create event (bulk RSVP creation for all active accounts on save)
- Edit event (name, date/time, type; type-change effects on RSVPs and fields per spec table)
- Delete event (cascades to all RSVPs)
- Custom fields for `other` events: add field (label, type, required, position; choices for `choice` type), edit field (blocked if responses exist), delete field (blocked if responses exist)
- RSVP on account activation: seed records for future events (inside invite flow transaction ŌĆö already in 3.3, but requires events table to exist)

---

## Phase 4 ŌĆö Data Migration

One-time operation. Run against the full schema once Phase 3 is complete.
See `specs/technical-specs/07-data-migration.md` for the detailed procedure.

- Pre-migration: encoding repair (mojibake ŌåÆ UTF-8), Google Sheet XLSX staging,
  referential integrity pre-check
- Migration steps 1ŌĆō9: seasons, instruments, accounts (OHM Agenda + Google Sheet enrichment),
  roles, fee payments, events (with ID offsets), RSVPs, sequence reset
- Admin role assignments (manual list from stakeholders)
- Verification queries (all 8 checks)
- Post-migration admin checklist: designate current season, review flagged accounts,
  send invite links to all active accounts

Sequenced delivery plan for `specs/goals/vision.md` § V1.

## Phase 0 — Walking Skeleton

**Status: Complete**

**Goal:** full vertical slice working locally. One DB table rendered in a browser. Proves
the stack (Go, Buffalo, Pop, PostgreSQL, Docker, Mise) hangs together before any feature work.

- Buffalo project initialised (`app.go`, `go.mod`, directory layout)
- Docker Compose: `postgres` container for local dev (the `app` container is a Phase 3 concern)
- `mise.toml` with `dev`, `stop`, `migrate`, `test`, `lint`, and other tasks
- `.env.development.example` (placeholders for `DATABASE_URL`, `SESSION_SECRET`, `GO_ENV`)
- First migration: `instruments` table (seeded, no FK dependencies)
- Instrument seed migration (OHM Agenda controlled list + "Chef d'orchestre")
- One public route `/instruments` → handler → Plush template listing instrument names
- `buffalo dev` starts, page loads data from DB
- Repository interfaces (DIP) wiring actions to models via `actions/repositories.go`

## Phase 1 — CI

**Status: Complete**

**Goal:** automated quality gate on every pull request; test infra established for all subsequent work.

- Lefthook (`lefthook.yml`): `gofmt`, `go vet`, `golangci-lint` on commit — registered
  automatically via `mise install` (`[prepare.lefthook]` + `[[hooks.postinstall]]`)
- Conventional commit message validation via `commit-msg` Lefthook hook
- `go mod tidy` drift check via pre-commit Lefthook hook
- `webapp/.golangci.yml`: pins v2 config format and explicit standard linter set
- `testcontainers-go` harness: `TestMain` spins up a PostgreSQL container, applies
  migrations, runs all tests in that binary, tears down
- Unit tests for actions (stub-based, no DB); integration tests for models (real PostgreSQL)
- GitHub Actions (`.github/workflows/ci.yml`): `lint`, `test`, and `migrations` jobs running
  in parallel on every pull request; all required to pass; Go module and build caches wired
- `migrations` job: spins up a PostgreSQL service container, runs up → down → up against
  `db/migrations/`; decoupled from Go tests — schema correctness, not application behaviour
- Coverage: per-package % in test job log + HTML artifact uploaded on every run;
  diff coverage (new code per PR) deferred to when it becomes useful

## Phase 2 — Auth Foundation

**Status: Next**

**Note:** architecture design contributed by a DDD review pass on the walking skeleton.

**Goal:** complete the walking skeleton with working session-based auth. A developer can
log in, reach a protected route, and log out. Validates the full auth stack before
production infra is built. Also establishes the three-layer architecture that all Phase 4
features must follow.

### Package architecture (establish in this phase, follow in all subsequent phases)

The walking skeleton has two layers: `actions/` (HTTP handlers) and `models/` (DB access).
Phase 4 features include complex domain operations (anonymization, invite flow, last-admin
protection, event type-change side effects) that belong in neither layer. A third layer is
introduced here, when `Account` — the first domain-rich entity — is implemented:

```
actions/    HTTP orchestration only. Thin: extract tx from context, call service or
            repository, set template variables, render. No business rules here.

services/   Domain logic. Business rules, complex transactions, cross-entity operations.
            Depends on repository interfaces (defined in this package). No direct SQL.
            Fully testable with stubbed repositories.

models/     DB access only. Structs with `db:` tags, Pop queries, repository interface
            implementations. No business rules here.
```

**Dependency direction:** `actions` → `services` → repository interfaces ← `models`.
`actions` never imports `models` directly after this phase.

**Repository interfaces** currently live in `actions/repositories.go` (defined on the
consumer side per DIP). As `services/` becomes the primary consumer of repositories,
interface definitions migrate there. The existing `InstrumentRepository` in
`actions/repositories.go` moves to `services/repositories.go` as part of this phase.

**Domain constants** — `AccountStatus`, `EventType`, `PaymentType`, `RsvpState` — are
typed string constants defined in `services/` (or a `domain/` sub-package), not raw
strings scattered across handlers and queries:

```go
type AccountStatus string
const (
    StatusPending    AccountStatus = "pending"
    StatusActive     AccountStatus = "active"
    StatusAnonymized AccountStatus = "anonymized"
)
```

**Testing convention:**
- `services/` tests: stub repositories (no DB required) — test business rules in isolation
- `models/` tests: real PostgreSQL via testcontainers — test queries and migrations
- `actions/` tests: stub services (no DB required) — test HTTP layer in isolation

### Unit 1 — Migrations

- [ ] `accounts` migration (full schema per `technical-specs/00-data-model.md` § accounts)
- [ ] `roles` + `account_roles` migrations; `admin` role seeded
      (see `technical-specs/00-data-model.md` § roles, § account_roles)
- [ ] `http_sessions` migration (custom schema: `id`, `key`, `data`, `created_on`,
      `modified_on`, `expires_on`, `account_id FK→accounts ON DELETE CASCADE`; index on

### Unit 2 — Services layer + auth logic

- [ ] `services/` package created; `InstrumentRepository` interface moved from
      `actions/repositories.go` to `services/repositories.go`; `actions/instruments.go`
      updated to import from `services/`
- [ ] Domain constants defined: `AccountStatus` in `services/`
- [ ] `AccountRepository` interface defined in `services/repositories.go`
- [ ] `AccountService` struct in `services/account.go` with:
      - `Authenticate(tx, email, password) (*models.Account, error)` — Argon2id verify;
        returns typed error distinguishing "not found" from "wrong password" (both presented
        identically to the user — no account enumeration)
      - `GetByID(tx, id) (*models.Account, error)`
- [ ] Unit tests for `AccountService.Authenticate` (stub repository, no DB)
- [ ] Integration tests for `AccountRepository` implementation (real PostgreSQL)

### Unit 3 — HTTP integration

- [ ] pgstore wired as session store in `actions/app.go` (replaces Buffalo's default cookie
      store); `SESSION_SECRET` env var consumed here
- [ ] Login form (`GET /connexion`) + POST handler (`POST /connexion`): calls
      `AccountService.Authenticate`, on success writes `account_id` to both session data and
      `http_sessions.account_id`, redirects to `/`
- [ ] Logout handler (`POST /deconnexion`): destroys session row, clears cookie, redirects
      to `/connexion`
- [ ] `RequireActiveAccount` middleware: reads `account_id` from session, calls
      `AccountService.GetByID`, checks `status == StatusActive`; redirects to `/connexion`
      on failure
- [ ] `RequireAdmin` middleware: after `RequireActiveAccount`, queries `account_roles` for
      admin role; returns 403 on failure
- [ ] One protected stub route (`GET /admin`) behind both middleware to prove the chain works
- [ ] `seed-admin` grift task (`mise run seed-admin`): creates first active admin account;
      refuses if one already exists; see `technical-specs/02-configuration.md` § Bootstrap
- [ ] `recover-admin` grift task (`mise run recover-admin`): force-resets password on any
      active account; see `technical-specs/02-configuration.md` § Emergency Admin Recovery
- [ ] Action tests for login/logout handlers (stub service)

## Phase 3 — Production Infra

**Goal:** application reachable at the OVH domain over HTTPS, daily backups running.

- Multi-stage `Dockerfile` (build stage: `golang:alpine`; runtime stage: `alpine`, ~25–30 MB image)
- `docker-compose.yml` extended with `caddy` service; `Caddyfile` with automatic HTTPS
- `.env.production.example`
- `mise run deploy` task: SSH to VPS → `docker compose pull` → `up -d` → `buffalo pop migrate`
- `DEPLOY_HOST` read from operator shell profile (not committed)
- `scripts/backup.sh`: `pg_dump` → gzip → upload to OVH Object Storage (30-day retention)
- VPS cron setup instructions for the backup script
- Production smoke test checklist (HTTPS green, login page loads, backup job runs once manually)

## Phase 4 — Features

Each section is an independently deployable unit. Implement in order — each section
depends on the ones above it.

---

### 4.1 Public Pages + Navigation Shell

Static pages and the navigation structure that wraps all subsequent features.

- Homepage template (placeholder copy; real copy provided by association before launch)
- Privacy notice template (content from `specs/goals/gdpr.md` §5)
- Layout template with auth-aware navigation:
  - Unauthenticated: login link
  - Authenticated: events, profile, sheet music (if configured), logout
- Footer with persistent privacy notice link (all pages)
- `SHEET_MUSIC_URL` env var wired to show/hide "Partitions" menu item

---

### 4.2 Account Lifecycle

Invite and password reset flows. Depends on Phase 2.

- `invite_tokens` + `password_reset_tokens` migrations
- Invite URL display in admin UI (after account creation — admin copies and sends manually)
- Invite form: password field, privacy acknowledgement checkbox, phone/address consent checkbox
- Invite flow completion (atomic transaction per `technical-specs/05-implementation-notes.md`):
  activate account, set password hash, set consent flag, clear phone/address if no consent,
  mark token used, seed RSVP records for future events, create session, redirect to `/evenements`
- Token regeneration (admin generates new invite link; invalidates existing token)
- Password reset: token generation + display, reset form, apply new password, mark token used
- Expired/used token handling (generic informative message, no account state change)

---

### 4.3 Musician Management

Full admin control over musician accounts, plus musician's own profile view. Depends on 4.2.

- Musician list (`/admin/musiciens`)
- Create musician account (with under-15 rule, invite token generated on save)
- Edit musician account (name, email, instrument, birth date, parental consent URI; phone/address gated on consent flag)
- Admin role grant / revoke (with last-admin protection check)
- Delete pending account (with last-admin protection check)
- Anonymization (atomic transaction: clear personal fields, remove roles, delete RSVPs,
  invalidate tokens, revoke session; last-admin protection check before transaction)
- Consent withdrawal: clear phone, address, and consent flag together
- Toggle processing-restricted flag (display only in V1)
- Musician's own profile view (`/profil`): name, instrument, email, birth date if set,
  phone/address if consented, static consent-withdrawal notice
- Retention review list (`/admin/retention`): accounts whose last-payment season ended > 5 years ago
- `seed-dev` grift task (implement once schema is complete enough to generate realistic data)

---

### 4.4 Season Management

Depends on Phase 2.

- `seasons` migration
- Season list (`/admin/saisons`) with inline creation form
- Create season (label, start date, end date; not automatically designated current)
- Designate current season (atomic swap transaction; exactly-one invariant maintained)

---

### 4.5 Fee Payments

Depends on 4.3 + 4.4.

- `fee_payments` migration
- Record fee payment on musician detail page (duplicate guard: reject if (account, season) pair exists)
- Edit fee payment (`/admin/cotisations/{id}/modifier`)
- Delete fee payment
- First inscription date: derived on demand (`MIN(payment_date)`), shown in admin musician detail view

---

### 4.6 Events + RSVP

Depends on 4.3 (accounts must exist for RSVP seeding).

- `events`, `rsvps`, `event_fields`, `event_field_choices`, `rsvp_field_responses` migrations
- Event list for authenticated users (`/evenements`): past 30 days + all upcoming, own RSVP state shown
- Event detail page: full RSVP list; for concerts, instrument per `yes` RSVP; for `other`, field responses per `yes` RSVP
- RSVP update (`/evenements/{id}/rsvp`): yes/no/maybe; instrument selection for concerts (main instrument pre-selected); response clearing on transition away from yes
- Admin event management list (`/admin/evenements`)
- Create event (bulk RSVP creation for all active accounts on save)
- Edit event (name, date/time, type; type-change effects on RSVPs and fields per spec table)
- Delete event (cascades to all RSVPs)
- Custom fields for `other` events: add field (label, type, required, position; choices for `choice` type), edit field (blocked if responses exist), delete field (blocked if responses exist)
- RSVP on account activation: seed records for future events (inside invite flow transaction — already in 4.2, but requires events table to exist)

---

## Phase 5 — Data Migration

One-time operation. Run against the full schema once Phase 4 is complete.
See `specs/technical-specs/07-data-migration.md` for the detailed procedure.

- Pre-migration: encoding repair (mojibake → UTF-8), Google Sheet XLSX staging,
  referential integrity pre-check
- Migration steps 1–9: seasons, instruments, accounts (OHM Agenda + Google Sheet enrichment),
  roles, fee payments, events (with ID offsets), RSVPs, sequence reset
- Admin role assignments (manual list from stakeholders)
- Verification queries (all 8 checks)
- Post-migration admin checklist: designate current season, review flagged accounts,
  send invite links to all active accounts

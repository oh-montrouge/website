# Epic EP-002: Account Privileges and Sheet Music Access

Status: draft

## Goal

Admins can grant and revoke the admin role on any account with mandatory last-admin protection enforced by the system; all authenticated users can reach the association's sheet music via a configured Google Drive link.

## Context

OHM musicians need differentiated access: any account can be elevated to admin to manage the association, but the system must ensure at least one admin always exists. Sheet music is the primary day-to-day value for authenticated musicians — the link that replaces the old shared-credential page must be immediately visible after login. Both capabilities are access-control decisions that sit on top of the session infrastructure built in EP-001.

## Personas

- **Admin**: An association member with management privileges. Uses the admin UI to grant or revoke the admin role on other musician accounts. Expects a clear error message when an operation is blocked by last-admin protection.
- **Musician**: An authenticated association member. Accesses the association's sheet music via a dedicated menu item when the Google Drive link is configured. Does not manage the link; only uses it.
- **System Operator**: A technical person deploying the application. Sets the `SHEET_MUSIC_URL` environment variable at deploy time. No UI interaction for this; it is a deploy-time configuration.

### Completion Criteria

An admin can grant the admin role to any active account and revoke it from any account that is not the sole remaining admin. The system rejects — with an informative message — any attempt to revoke the admin role when the target account is the last active admin. When `SHEET_MUSIC_URL` is configured at deploy time, all authenticated users see a "Partitions" menu item in the navigation; the item is absent when the variable is unset. Unauthenticated users cannot see the menu item.

### Non-Functional Requirements

- NFR-000-1: Admin role must be re-verified from the database on every request — not from cached session state — so that a revoked admin loses access immediately on their next request.
- NFR-000-2: The last-admin invariant ("at least one active account holds the admin role at all times") is enforced at the application layer. It is not enforced by a database constraint.
- NFR-000-3: The sheet music URL is a deploy-time environment variable (`SHEET_MUSIC_URL`). No admin UI exists to manage it.
- NFR-000-4: All UI text and error messages are in French.

### Related External Components

- **Component C-001 — Google Drive**: Hosts the sheet music files. The application provides only a hyperlink; file access control is managed entirely at the Google Drive level. The application has no visibility into whether the musician can actually access the files.

### Out of Scope

- **Creating, editing, or deleting musician records** — ep-2 scope.
- **Account anonymization** (which removes all roles from the account) — ep-4 scope. The last-admin protection check that applies during anonymization is ep-4's responsibility to implement, guided by the invariant defined here.
- **Deleting a pending admin account** — ep-2 scope. The last-admin protection check for that operation is ep-2's responsibility to implement, guided by the invariant defined here.
- **Additional role types beyond `admin`** — V1 defines a single role; the schema supports future roles but no further role-type UI is in scope.
- **Sheet music file management, hosting, or search** — vision "Later" feature.
- **Admin UI to configure the sheet music URL** — by design; deploy-time configuration only.
- **Emergency admin recovery CLI** — EP-001 CAP-002 scope.

### Assumptions

- **ASM-000-1**: The last-admin protection invariant applies to three operations: (1) revoking the admin role, (2) anonymizing an admin account, (3) deleting a pending admin account. *Why*: The functional spec enumerates exactly these three operations under "Last-Admin Protection." Only (1) is implemented in this epic; (2) and (3) are implemented in ep-4 and ep-2 respectively, but they must honour the same invariant. — ⚠️ Confidence: HIGH
- **ASM-000-2**: A pending account can hold the admin role (set by an admin before the account is activated), and last-admin protection blocks its deletion if it is the sole admin. *Why*: The functional spec explicitly lists "Deleting a pending admin account" as one of the blocked operations, implying the role can be set on a pending account. — ⚠️ Confidence: HIGH
- **ASM-000-3**: An admin can grant the admin role to themselves (for use during initial setup before other accounts are created). *Why*: The spec places no restriction on self-grant; the bootstrap CLI creates the first admin account, and subsequent grant/revoke is via the admin UI. — ⚠️ Confidence: MEDIUM

None. All behavioral details are specified in the source material.

## Capability CAP-001 - Admin Role Management and Last-Admin Protection

Admin can grant or revoke the admin role on any account, with the system blocking any operation that would leave the association without a remaining active admin.

### References

- Functional spec: `specs/functional-specs/01-account-lifecycle.md` — Last-Admin Protection section
- Functional spec: `specs/functional-specs/00-information-model.md` — Account entity (Roles, Role invariant)
- Technical spec: `specs/technical-specs/01-auth-and-security.md` — Role Checks section
- Technical spec: `specs/technical-specs/04-routing.md` — `/admin/musiciens/{id}/role-admin` routes (POST grant, DELETE revoke), `RequireAdmin` middleware
- Technical spec: `specs/technical-specs/00-data-model.md` — `roles`, `account_roles` tables, application invariant note
- Technical ADR: `specs/technical-adrs/005-role-model.md`

### Description

From the admin musician-management UI, an admin can grant the admin role to an active account or revoke it from an account. The system queries the `account_roles` table on every request to verify admin status — not from session state — ensuring a revoked admin loses access on their next page load. Any attempt to revoke the admin role from the sole remaining active admin is rejected with an informative message. This capability defines the invariant that governs last-admin protection; the checks for anonymization and pending-account deletion are implemented in ep-4 and ep-2 respectively but honour the same rule.

### Depends on

- Capability EP-001 CAP-002 — Login, Logout, and Session Management: The `RequireAdmin` middleware runs after `RequireActiveAccount`; a valid active-account session is required before the role lookup.

### Out of Scope

- Last-admin protection during account anonymization — ep-4 scope.
- Last-admin protection during pending account deletion — ep-2 scope.
- Granting or revoking roles other than `admin` — no additional roles in V1.
- Any role management UI beyond the grant/revoke actions on the musician detail view.

### Assumptions

- **ASM-001-1**: The admin role check counts only `active` accounts with the `admin` role. A pending account holding the admin role does not count toward "at least one active admin." *Why*: The functional spec states "at least one `active` account with the admin flag set" and the last-admin protection blocks operations when they "would result in zero active admin accounts." — ⚠️ Confidence: HIGH
- **ASM-001-2**: The grant and revoke actions appear on the musician detail view (`/admin/musiciens/{id}`), not as a separate management screen. *Why*: The routing spec defines `POST /admin/musiciens/{id}/role-admin` and `DELETE /admin/musiciens/{id}/role-admin` without a separate list page for role management. — ⚠️ Confidence: HIGH

## Capability CAP-002 - Sheet Music Access

Authenticated users see a "Partitions" menu item in the application navigation linking to the configured Google Drive sheet music URL; the item is hidden when no URL is configured.

### References

- Functional spec: `specs/functional-specs/01-account-lifecycle.md` — Sheet Music Access section
- Technical spec: `specs/technical-specs/02-configuration.md` — Environment Variables (`SHEET_MUSIC_URL`)
- Goal spec: `specs/goals/vision.md` — "Musician can access sheet music via a single Google Drive link, configured at deploy time"

### Description

When the `SHEET_MUSIC_URL` environment variable is set at deploy time, a "Partitions" menu item appears in the navigation bar for all logged-in users. Clicking it opens the configured Google Drive URL. If `SHEET_MUSIC_URL` is not set, the menu item is absent from the navigation entirely. Access control for the files themselves is managed at the Google Drive level; the application has no role in enforcing it.

### Depends on

- Capability EP-001 CAP-002 — Login, Logout, and Session Management: The menu item is displayed only to authenticated users; session management must exist before the visibility rule can be enforced.

### Out of Scope

- Admin UI to configure the sheet music URL — deploy-time environment variable only.
- Application-level access control for the sheet music files — Google Drive handles this.
- Sheet music file hosting, indexing, or integrated search — vision "Later" feature.
- Any menu item for unauthenticated users or for users who have not yet completed the invite flow.

### Assumptions

- **ASM-002-1**: The "Partitions" menu item label is the French word "Partitions" (sheet music). *Why*: The application UI is in French; "partitions" is the direct French equivalent of "sheet music." — ⚠️ Confidence: HIGH
- **ASM-002-2**: The menu item opens the configured URL in a new browser tab (or follows it directly). *Why*: The spec says the item "opens the configured Drive URL" without specifying tab behaviour; opening in a new tab is conventional for external links. The story writer decides the exact behaviour. — ⚠️ Confidence: MEDIUM

None.

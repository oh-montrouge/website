## Environment Variables

| Variable | Required | Notes |
|----------|----------|-------|
| `DATABASE_URL` | Yes | PostgreSQL connection string (e.g. `postgres://user:pass@host/ohm?sslmode=disable`) |
| `SESSION_SECRET` | Yes | Secret for signing session cookies. Minimum 32 random bytes. Keep private. |
| `GO_ENV` | Yes | Runtime environment: `development` \| `test` \| `production` |
| `PORT` | No | HTTP server port. Default: `3000`. |
| `SHEET_MUSIC_URL` | No | Google Drive link for sheet music. If set, the "Partitions" menu item is shown to authenticated users. If absent, the menu item is hidden. |

## Bootstrap: First Admin Account

The system provides a CLI command to create the first administrator account outside the normal
invite flow.

**Behaviour:**
- Creates an account in `active` state and grants it the `admin` role.
- Accepts email and password as arguments (or interactive prompt).
- Refuses to run if an `active` admin account already exists, to prevent accidental duplication.
- The password is hashed with Argon2id before storage; it is never echoed or logged.

**Implementation:** A `buffalo task` (grift task) defined in `grifts/db.go`, invoked via
`mise run seed-admin` (wraps `buffalo task db:seed:admin`). See [CI/CD spec](06-ci-cd.md).

**Usage context:** One-time setup after first deployment. Not exposed as an HTTP endpoint.

## Emergency Admin Recovery

For use when the last admin cannot log in (forgotten password, corrupted credential). Bypasses
the normal password reset flow, which requires an admin to already be logged in.

**Behaviour:**
- Accepts an email address and a new password as arguments (or interactive prompt).
- Verifies the account exists with `status = 'active'`.
- Generates a new Argon2id hash and writes it directly to `accounts.password_hash`.
- Does not check for admin role — this is intentional (it is a recovery tool, not a
  management operation).
- The password is hashed before storage; it is never echoed or logged.

**Implementation:** Grift task in `grifts/db.go`, invoked via `mise run recover-admin`
(wraps `buffalo task db:recover:admin`).

**Usage context:** Emergency recovery only. Requires SSH access to the VPS and Docker
running. Never exposed as an HTTP endpoint.

## Development Seed Data

A separate grift task populates the local database with realistic dummy data for development:
several musicians in varied states (pending, active), a current season, past seasons, events
of each type with RSVPs, and fee payment records.

**Implementation:** `grifts/db.go`, invoked via `mise run seed-dev` (wraps
`buffalo task db:seed:dev`).

**Usage context:** Local development only. Never run against production.

## Instrument Seed Data

The instrument list is fixed at migration time. A seed migration populates the `instruments`
table from the OHM Agenda controlled list, plus "Chef d'orchestre". No admin UI exists to add
or remove instruments in V1.

The list is extracted from OHM Agenda at migration time (`current/ohm-agenda/db_schema.sql`
or `db_export.sql`). "Chef d'orchestre" is not present in the OHM Agenda export; the migration
adds it explicitly as an additional row.

## Season Invariant and Initial State

The functional spec requires exactly one season to be designated current at all times. This
invariant applies in steady state (once any season has been created and designated current).

Before the first season is created, the system has no current season. This is an acceptable
transient state during initial setup. The application does not enforce the invariant on an empty
`seasons` table; it enforces it on any operation that would remove the current designation from
an existing season (transferring it to another).

**Recommended setup sequence after first deployment:**
1. Run the bootstrap CLI to create the first admin account.
2. Log in and create the initial season.
3. Designate the initial season as current.

No seed season is created by migrations; real seasons are created by the admin.

## Schema Versioning

DB schema changes are managed through Pop migrations (Fizz DSL or raw SQL), located in
`migrations/`. Each migration file has an `up` and `down` direction.

**Commands:**
- `buffalo pop migrate` — apply all pending migrations
- `buffalo pop migrate --rollback` — roll back the last applied migration
- `buffalo pop migrate status` — list applied and pending migrations

Conventions:
- One migration file per logical schema change.
- Migration files are named with a timestamp prefix (generated by `buffalo pop g model`).
- Migrations are applied in order and never modified after deployment.
- Pop tracks applied migrations in a `schema_migration` table.
